TL;DR: This post is about Cloud Run and Scheduling a call of a Cloud Run service with Cloud Scheduler. Precisely we use Cloud Run to deploy a stateless birthday reminder app and use Cloud Scheduler to call the deployed service API.

In the previous post, I deployed a simple birthday reminder app on Google Cloud Functions and scheduled a cron job using Cloud Scheduler to invoke the app every day. Upon invocation, the app checks for birthdays and sends out an SMS to your configured phone number if there is at least one birthday that day. It works fine. Today, I am going to package up the birthday app in a container image using Docker, push it to Google Container Registry and then deploy it to Cloud Run.

1. What is Docker

Docker is a platform that allows you to package up your application and its dependencies in a container image. Essentially it allows you to build, share and run container images.

2. What is a container image

It's the image of a container. It's built from a file called Dockerfile, which is a script-like file that contains a series of commands which are to be executed successively by docker to create a container image.

A container image is your app bundled up together with all the dependencies that are necessary to make it run. You can package 'anything' in a container image. Any runtime, any binary, any library. Instead of container image, often we just say image.

3. What is a container

A container is a running image. Basically when your app packaged up in a container image is spun up (executed), it's called container.

4. What is Container Registry

Container Registry is Google's repository for container images. Put simply, it's like the GitHub of container images, however it's not public. The largest public repository for container images is Docker Hub. With Google Container Registry, you can manage your Docker images, perform vulnerability analysis, and because it's a GCP service, you decide who can access what with fine-grained access control using IAM policies.

5. What is Cloud Run

Cloud Run is the Cloud Functions for containers. It's a fully managed serverless compute solution that enables you to run any stateless application. It is really flexible, since you can code up an app using any runtime and any library, package it up in a container image and run it on Cloud Run. It's not limited to a couple of runtimes as it is the case with Cloud Functions. But similar to Cloud Functions, Cloud Run can scale down to zero when it's not being used, thereby costing nothing. It can also scale up to handle increasing user requests.

# process_calendar.py
from icalendar import Calendar
from collections import namedtuple
from datetime import date, timedelta
import sqlite3

birth_day = namedtuple('BD', ['name', 'month', 'day'])

def read_calendar():
    with open('cal.ics', 'rb') as f:
        calics = Calendar.from_ical(f.read())
        for component in calics.walk(name='VEVENT'):
            name = component.get('SUMMARY').replace("'s birthday", "")
            birth_date = component.get('DTSTART').dt
            # print(name, birth_date.strftime("%b %d"))
            yield birth_day(name=name,
                            month=str.lower(birth_date.strftime("%b")),
                            day=birth_date.day)


def persist_data():
    connection = sqlite3.connect('birthdays.db')
    cursor = connection.cursor()
    cursor.execute(
        '''CREATE TABLE IF NOT EXISTS birthdays
        (id INTEGER NOT NULL PRIMARY KEY AUTOINCREMENT,
        name INTEGER NOT NULL,
        month TEXT NOT NULL,
        day INTEGER NOT NULL)''')
    for item in read_calendar():
        cursor.execute('''INSERT INTO birthdays VALUES (NULL,?,?,?)''',
                       (item.name, item.month, item.day))
    connection.commit()
    connection.close()


if __name__ == '__main__':
    persist_data()

The script outputs a file, birthdays.db, which is essentially the birthday data in a sqlite database.

# birthdays.py
from datetime import date, timedelta
import sqlite3

def fetch_birthday_friends_by_date(month, day):
    connection = sqlite3.connect('birthdays.db')
    c = connection.cursor()
    f_list = []
    for row in c.execute('''SELECT DISTINCT name FROM birthdays
							WHERE month=? and day=?''',
                         (month, day,)):
        f_list.append(row[0])
    connection.close()
    return "{} {}".format(month, day), f_list

# app.py
import os
from datetime import date
from flask import Flask, request, jsonify
from twilio.rest import Client
import birthdays as bdays

app = Flask(__name__)

def send_sms(msg, from_='+166...', to='+233...'):
    # Your Account Sid and Auth Token from twilio.com/console
    account_sid = ''
    auth_token = ''
    client = Client(account_sid, auth_token)
    message = client.messages.create(body=msg, from_=from_, to=to)
    print(message.sid)

@app.route('/')
def index():
    return "<h1>Hello Containerizers!!</h1>"

@app.route('/birthday_friends', defaults={'month': None, 'day': None})
@app.route('/birthday_friends/<month>-<int:day>')
def get_birthday_friends(month, day):
    if not month or not day:
        now = date.today()
        month = str.lower(now.strftime("%b"))
        day = now.day
    birthdate, f_list = bdays.fetch_birthday_friends_by_date(month, day)
    bday_friends = {}
    bday_friends['date'], bday_friends['names'] = birthdate, []
    if f_list:
        bday_friends['names'] = f_list
        if request.args.get('notify') == 'true':
            send_sms(bday_friends)
return jsonify(bday_friends)

if __name__ == '__main__':
    app.run(host="0.0.0.0", port=int(os.environ.get('PORT', 8080)), debug=True)

7.2. Tell Cloud Build to do everything

In this section, we are going to use Cloud Build to build and tag a Docker image, push the to Google Container Registry and deploy it to Cloud Run. Cloud Build is a Google's service that offers CI/CD fonctionalities that cover all the software development stages in cloud environments as well as on premises environments. Essentially you submit the build, test and deployment processes to Cloud Build by providing the build steps in a configuration file. The builds run on the GCP and you can use high-CPU VMs to further increase the build speed.

We will be using Cloud SDK gcloud command-line tool If you haven't installed it yet, go ahead and install it. If you have already installed it, run gcloud components update to have the latest available version.

Run this command to log in to GCP

1gcloud auth login

Set the project that you will be working with. Replace PROJECT_ID by the actual project ID.

1gcloud config set project PROJECT_ID

Export the project ID as the environment variable $GCLOUD_PROJECT; it will save us from the mechanical task of typing and typos.

1export GCLOUD_PROJECT=`(gcloud config get-value project)`

You can list all the projects you have access to with the current account by running gcloud projects list.

Let's also export the project number, because we are going to use it multiple times.

1export PROJECT_NUMBER=`(gcloud projects describe $GCLOUD_PROJECT --format="value(projectNumber)")`

Enable APIs

To build container images on the GCP and deploy them to Cloud Run, we have to enable the Resource Manager API, Cloud Build API, the Container Registry API and Cloud Run API:

1gcloud services enable cloudbuild.googleapis.com \
2    containerregistry.googleapis.com \
3    cloudresourcemanager.googleapis.com \
4    run.googleapis.com

Build and run

Now let's build a docker image and push it to Container Registry with the command below. 

1gcloud builds submit --tag gcr.io/$GCLOUD_PROJECT/bdays-reminder:0.1

Let's deploy the container image we just built to Cloud Run 

1gcloud beta run deploy bdays-reminder-service-m \
2   --image gcr.io/$GCLOUD_PROJECT/bdays-reminder:0.1 \
3   --region us-central1 --platform managed --memory 128Mi \
4   --timeout 60 --allow-unauthenticated

Now, you are probably thinking, didn't you write "Tell Cloud Build to do everything", but Cloud Build didn't actually do everything, we just manually deployed a container image to Cloud Run. You are right. We tell Cloud Build to do everything in the section below.

IAM permissions

Before we start the build, let's talk (think) about IAM permissions.

If we were using Cloud Console, we wouldn't have any problem with the build and deployment. In fact when we previously ran gcloud builds and gcloud beta run deploy commands we didn't worry about permissions. Why? Because as project owner, you have all the permissions to do everything to your project including screwing everything up. But since we are going to use Cloud Build to deploy our container image to Cloud Run, Cloud Build service account has to be granted Cloud Run Admin role, else it can't deploy to Cloud Run. The same thing apply to other GCP services. For Cloud Build to interact with other GCP services, we have to grant Cloud Build service account the necessary roles so it has the permission to perform the tasks.

With the command below, we are adding an IAM policy binding for the role of roles/run.admin to the Cloud Build service account which is $PROJECT_NUMBER@cloudbuild.gserviceaccount.com. Remember that we exported the actual project number as the env var $PROJECT_NUMBER.

1gcloud projects add-iam-policy-binding $GCLOUD_PROJECT \
2      --member="serviceAccount:$PROJECT_NUMBER@cloudbuild.gserviceaccount.com" \
3      --role='roles/run.admin'

We have to pay attention to another thing which is that a service account is also a resource besides being an identity. So as a GCP resource, they have IAM policies. A service account has to have the Service Account user role to be able to run operations as a service account ü§Ø. In our case, we have to grant the Service Account User role (roles/iam.serviceAccountUser) for Cloud Run service account to the Cloud Build service account. We will grant it at the service account level, not at the project level. Wait whaaa? :confused:.

Now let's run the command that will grant the service account user role to the service account. We know the Cloud Build service account identifier. But what about Cloud Run's? Cloud Run uses the Compute Engine default service account by default ($PROJECT_NUMBER-compute@developer.gserviceaccount.com). However it's not recommended to work with this default setup, because the Compute Engine default service account have read and write access to all resources in your GCP project. That's too much power. Good IAM policies should always follow the principle of least privilege. But let's go with the default for now.

1gcloud iam service-accounts add-iam-policy-binding \
2	$PROJECT_NUMBER-compute@developer.gserviceaccount.com \
3	--member="serviceAccount:$PROJECT_NUMBER@cloudbuild.gserviceaccount.com" \
4	--role='roles/iam.serviceAccountUser'

8. Create a con job to invoke the Cloud Run service

We just need to to create a job to periodically call the HTTP endpoint of the service we just deployed. Get the URL and replace it in the commands below:

1gcloud scheduler jobs create http bday-reminder-job \
2   --schedule="0 17 * * *" \
3   --uri="YOUR_URL/birthday_friends&notify=true" \
4   --http-method=GET

When we were deploying the image to Cloud Run, we used the --allow-unauthenticated flag, which makes Cloud Run services plublic, accessible to anybody who knows the HTTP endpoints. If we wanted to protect it we'd have used the --no-allow-unauthenticated flag instead. And in that case, to invoke the secured endpoint using Cloud Scheduler, we'd have to pass the authentication token to the header of the request like in the command below:

1gcloud scheduler jobs create http bday-reminder-job  \
2   --schedule="0 17 * * *"  \
3   --uri="$SERVICE_URL/birthday_friends&notify=true"  \
4   --http-method=GET \
5   --time-zone="Africa/Accra" \
6   --headers="{'Authorization'='Bearer `(gcloud auth print-identity-token)`'}"
